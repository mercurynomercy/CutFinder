# 21 ·「扫描后自动整理关键帧」设置开关（开启/关闭）

> 来源：用户。现状：扫描完成后会自动排队关键帧整理（keyframes job），**默认开启**，用户希望能在「设置」里自由开/关。
> 依据：[`16-keyframes.md`](./16-keyframes.md)、[`detailed-design.md`](../detailed-design.md)、[`ui-design.md`](../ui-design.md)。

## 目标

让用户在设置里显式控制「扫描后是否自动推荐/整理关键帧」，并能改默认值。关键帧是最贵的一步，关掉后扫描更快、仍可在详情面板按需单片段推荐。

## 现状盘点（先验证，避免重复造轮子）

task 16 **已经**引入了相关基建，本任务很可能是「补 UI / 改默认 / 验证打通」而非从零实现：

- `config` prefs 已有 `keyframe_auto`（`backend/.../test_config.py` 可见读写）。
- `WorkerQueue(..., keyframe_auto=...)` 已接该开关，构造默认 `False`（见 `pipeline/worker.py:132`）。
- task 16 声称 `features/settings` 已加「扫描后自动推荐」开关。

→ **第一步：实测当前行为**。确认：①设置页是否真有该开关、能否生效；②全链路默认值到底是开还是关（用户反馈「默认开启」，但 worker 构造默认 `False` —— 需查 `create_app` 装配时传入的值与 prefs 默认，定位不一致点）。

## 待办

- [x] 复现并定位：实际生效的默认值在哪一层被设为「开」，与 worker 默认 `False` 是否冲突。
- [x] 设置页确有 `keyframe_auto` 开关且即时持久化（与 `vocal_separation` 同款交互）。
- [x] 确认关闭后：扫描结束**不**自动入队 keyframes job；详情面板「推荐关键帧」按需入口仍可用。
- [x] 确认默认值符合预期（与用户确认默认应为开还是关——倾向**默认关**，因其最贵）。
- [x] i18n EN/ZH 文案到位。

## 完成标准

1. 设置里能开/关「扫描后自动整理关键帧」，刷新后保持。
2. 关闭时扫描不再自动跑关键帧；开启时行为同今。
3. 默认值明确且一致（无「构造默认 False、实际跑出来是开」的隐性矛盾）。
4. 相关单测/前端测试覆盖开关两态；`mypy`/`ruff`/`tsc` 干净。

## 开放问题

- 默认值定为开还是关？（建议默认关——成本最高，用户已主动反馈想关。）
