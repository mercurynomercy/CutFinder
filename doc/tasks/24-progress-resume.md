# 24 · 刷新页面后进度条恢复（dashboard 扫描 + 字幕导出）

> 来源：用户。Bug：刷新浏览器页面后，dashboard 的扫描进度条、以及字幕导出的进度条**消失了**；后台 job 其实还在跑，只是 UI 不再显示。希望刷新后能重新接上正在进行的进度。
> 依据：[`12-worker-queue.md`](./12-worker-queue.md)（队列 + SSE）、[`19-subtitle-progress.md`](./19-subtitle-progress.md)（字幕进度）、[`13-api.md`](./13-api.md)、[`14-frontend.md`](./14-frontend.md)、[`ui-design.md`](../ui-design.md) §6。

## 现象与根因（待确认）

进度状态目前**只活在前端内存里**——靠发起操作时拿到的 `job_id` + 后续 SSE 推送驱动进度条。刷新后内存丢失、SSE 订阅断开，前端不知道「还有 job 在跑」，于是进度条消失，尽管后端 job 仍在执行（worker 串行处理，状态在 DB/内存队列里）。

## 目标

页面加载/刷新时，前端能**主动发现**当前正在进行的 job 并**重连其进度**，让 dashboard 扫描进度条与字幕导出进度条恢复显示、继续平滑推进，直至完成。

## 待办

- [x] 后端：提供「列出活跃/未完成 job」的查询端点（按 kind：`scan` / `subtitle` / 其他），返回 `job_id`、`kind`、`done`/`total`、阶段标签等足以重建进度条的字段。（确认 `12`/`13` 是否已有 `GET /api/jobs` 或类似；缺则补。）
- [x] 后端：确认 SSE 端点支持**按 job_id 重新订阅**已在跑的 job（而非只在创建时推一次）；进度事件含当前累计值，便于中途接入。
- [x] 前端（dashboard）：挂载时查活跃 `scan` job → 若有则恢复进度条并重订阅 SSE。
- [x] 前端（字幕导出页）：挂载时查活跃 `subtitle` job → 恢复真实进度条 + 阶段标签（分离/转写，复用 task 19 的渲染）。
- [x] 进度条到达终态后正确收尾/清除；无活跃 job 时不显示。
- [x] 处理边界：刷新瞬间 job 刚好完成 / 失败 → UI 给出最终态而非卡住。

## 完成标准

1. 扫描或字幕导出进行中刷新页面 → 进度条**自动恢复**并继续推进到 100%，无需重新发起操作。
2. 没有进行中 job 时，刷新后不残留空进度条。
3. job 在刷新期间完成/失败 → UI 正确显示终态。
4. 后端「活跃 job 查询 + 按 id 重订阅 SSE」有单测；前端有「挂载时恢复进度」组件测试；`mypy`/`ruff`/`tsc` 干净。

## 开放问题

- job 状态是否已持久化到 DB（重启后端也能恢复），还是仅内存队列（仅同进程内可查）？这决定「刷新前端」与「重启后端」两种恢复能力的边界——本任务先保「刷新前端」即可。
