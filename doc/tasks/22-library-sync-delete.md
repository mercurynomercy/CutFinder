# 22 · 库文件删除 → 同步清理（DB 记录 + 缩略图 + 关键帧）

> 来源：用户。诉求：当用户在 **library（归档副本）** 里手动删掉某个文件，CutFinder 应同步把该片段在 SQLite 的记录、缩略图、关键帧等衍生物一并清掉，不要留下指向已不存在文件的「幽灵」条目。
> 依据：[`detailed-design.md`](../detailed-design.md)、[`09-catalog-repository.md`](./09-catalog-repository.md)、[`10-scanner.md`](./10-scanner.md)。

## 目标

让目录与目录状态收敛：库里副本被删 → catalog 里对应 clip 及其衍生物（缩略图、关键帧帧目录、transcript、tags、keyframes 等）级联清除。

## 硬约束（牢记 CLAUDE.md）

- **源文件只读**：本任务只针对 **library 内的副本**与 `.cutfinder/` 衍生物；**绝不**因为同步而去碰用户原始素材。
- **不误删**：只清理「确认副本已不在库」的条目；不能因为一次扫描读不到（如外接盘未挂载、权限抖动）就把数据删掉——需防御「整库不可达」的情况。
- 删除应**幂等**，与既有「重扫只处理新文件」的语义共存。

## 待定方案（两条路线，需先选）

1. **重扫时对账（reconcile-on-scan）**：在扫描流程末尾比对 catalog 里指向该库的 clip 与磁盘现状，对「记录在册但副本已消失」者清理。简单、无需常驻；缺点是需用户触发扫描才收敛。
2. **手动「清理失效条目」按钮**：设置/库管理页一个动作，扫一遍做对账并报告将删除的数量，确认后执行。最稳（有预览+确认），无误删风险。

> 倾向：先做 **路线 2（带确认的手动对账）**，或在路线 1 基础上加「整库不可达则跳过、不删」的保护。实时文件系统监听（FSEvents/watchdog）成本高、易抖动，**不在 v1**。

- [x] i18n EN/ZH。

## 完成标准

1. 删掉库里某副本后，经对账（自动或手动）→ 该 clip 记录、缩略图、关键帧、衍生标签/转写全部消失，画廊不再显示幽灵卡片。
2. 源素材始终未被触碰。
3. 库不可达时不发生误删（有保护 + 提示）。
4. 重复对账幂等；单测覆盖「副本消失→清理」「库不可达→跳过」「级联删除衍生物」；`mypy`/`ruff`/`tsc` 干净。
